现状分析 ​

业务场景分析 ​

OIM产品中，基于过车消费的业务场景比较简单：从kafka中消费xfusion[2]保存入库的过车数据（vehicle），业务组件进行二次消费处理，主要是用于实时预警相关业务。

图中客户端包括：实时预警、站点值守、报警助手、手机APP，可视化预警五个模块，但是整体接收的数据协议都是基于vehicle消息报文。

现有代码设计思路 ​

设计初衷 ​

现有的代码思路是不同的预警类型数据进行单独消费处理，也有可能是因为不同模块由不同的同事开发，所以都是自顾自写各自的消费逻辑，通过定义不同的Kakfka GroupId进行消费。

本身数据的消费是基于一个“管道模型”进行实现（最初参考了xtransfer[3]逻辑，基于ActiveMQ消费时候设计）：即将数据通过异步线程消费缓存到应用缓存中，基于注入此缓存管道的具体业务处理器再进行异步消费，而这样设计的初衷是当管道内的数据发生堆积，消费能力不足的时候进行主动的停止消费。

那么以数据监测的实时数据接入为例，大概看下代码结构吧。

问题分析 ​

📣声明 ​

以上代码并非我写，所以暂不对具体业务代码好坏与逻辑做评价。

这个模型最初是为ActiveMQ消费设计的，其实在Kafka消费中其实存在严重缺陷隐患（具体问题可参考记一次kafka不合理使用导致CPU过载的排查思路及解决办法），会导致CPU过载。

而且此模型将数据缓存到了应用层，同事实现新的消费一般都是拷贝粘贴的方式，没有根据实际业务需求修改内部的队列大小以及线程数等参数，在一定程度上极大增加了应用内存及线程的占用。另外还有一个问题是对于一份消息协议未做到统一的解析处理，增加了初始数据解析处理成本。

当然对此我负有主要责任，因为大伙儿基本都是拷贝了我以前的实现进行扩展开发，我本身设计的逻辑存在很大的局限性和安全隐患。不过还好，最终并未酿成大祸。

总体设计思路 ​

总体设计 ​

为了解决已知的问题，即需要摈弃之前的管道缓存模式；然后过车数据协议进行统一的数据解析分发；对于数据解析处理环节进行分步抽象（模板模式[4]），规范整体的数据处理流程。

数据消费监听到业务处理的过程中，以数据流来说是一个由抽象-》具体的过程；而反之业务处理到数据消费监听的过程则是一个由繁入简的过程。

模板模式 ​

将消息的处理过程通过TransactionBatchHandle接口抽象为：start、parseMessage、handleMessage、end四个环节；然后定义通用的处理模板类TransactionBatchDirector规范数据处理过程。

具体业务消费实现接口和注入模板即可实现统一的数据消费处理过程。

TransactionBatchHandle流水处理接口 ​

java
/**
 * 流水处理
 *
 * @author yuanzp
 * @date 2020/3/3 11:04
 * @digression done is better than perfect.
 */
public interface TransactionBatchHandle<M, R> {

    /**
     * 开始，消息日志输出之类操作（默认基类实现）
     *
     * @param message
     */
    void start(List<M> message);

    /**
     * 解析消息
     *
     * @param message 收到的消息
     * @return 处理后的消息
     */
    List<R> parseMessage(List<M> message);

    /**
     * 处理消息
     *
     * @param rMessage
     */
    void handleMessage(List<R> rMessage);

    /**
     * 结束，消耗时长等操作（默认鸡肋实现）
     *
     * @param startTime
     */
    void end(Date startTime);

}

TransactionBatchDirector流水处理模板 ​

java
/**
 * 流水处理模板
 *
 * @author yuanzp
 * @date 2020/3/3 11:12
 * @digression done is better than perfect.
 */
@Slf4j
public class TransactionBatchDirector<M, R> {
    private TransactionBatchHandle<M, R> transactionBatchHandle;

    public TransactionBatchDirector(TransactionBatchHandle transactionBatchHandle) {
        this.transactionBatchHandle = transactionBatchHandle;
    }

    /**
     * 消息的消费
     *
     * @param message
     */
    public void construct(List<M> message) {
        Date startTime = new Date();
        transactionBatchHandle.start(message);
        List<R> rMessage = transactionBatchHandle.parseMessage(message);
        if (rMessage == null) {
            return;
        }
        transactionBatchHandle.handleMessage(rMessage);
        transactionBatchHandle.end(startTime);
    }
}

OverrunRecordVehicleTransactionDeal超限数据处理器 ​

java
/**
 * 超限数据处理器
 * <p>
 * 统一接收超限过车消息进行重构 {@link #handleMessage(List)} ，处理成为对象 {@link RealTimeRecord} ，
 * 在 {@link #handleMessage(List)} 中进行统一分发，
 * 即通过实现实现目标接口 {@link IMessageConsumer} 即可实现过车消费能力。
 * 目前已有的消费类如下：
 * <ol>
 *     <li>实时预警 {@link OverrunRecordListener}</li>
 *     <li>站点值守 {@link DutyOnStationListener}</li>
 *     <li>报警助手 {@link AlarmMessageForAlarmToolConsumer}</li>
 *     <li>手机APP {@link AppOverrunMessageProducer}</li>
 *     <li>简版可视化 {@link VisualOverrunRecordListener}</li>
 * </ol>
 * </p>
 * @author yuanzp
 * @since V1.3.0
 */
@Slf4j
@Component
public class OverrunRecordVehicleTransactionDeal extends BaseTransactionHandle implements TransactionBatchHandle<ConsumerRecord<?, ?>, OverrunRecordWarnDTO> {
    private ExecutorService executorService;
    private List<IMessageConsumer<RealTimeRecord>> iMessageConsumerList;
    @Autowired
    private RealTimeRecordAdapter realTimeRecordAdapter;
    @Autowired
    private TransactionHandleHelp transactionHandleHelp;
    @Autowired
    private OverrunThresholdService overrunThresholdService;

    public OverrunRecordVehicleTransactionDeal(List<IMessageConsumer<RealTimeRecord>> iMessageConsumerList) {
        this.iMessageConsumerList = iMessageConsumerList;
        int size = iMessageConsumerList.size();
        this.executorService = new ThreadPoolExecutor(size, size + 1, 1L,
                TimeUnit.SECONDS, new LinkedBlockingQueue<>(size),
                Executors.defaultThreadFactory(), new ThreadPoolExecutor.AbortPolicy());
    }

    @Override
    public List<OverrunRecordWarnDTO> parseMessage(List<ConsumerRecord<?, ?>> message) {
        List<OverrunRecordDTO> overrunRecordDTOS = Optional.ofNullable(message)
                .map(transactionHandleHelp::consumerRecordGetMessage)
                .filter(transactionHandleHelp::checkBlank)
                .map(transactionHandleHelp::parseOverrunRecordDTOs)
                .orElse(CollUtil.newArrayList());
        List<OverrunRecordWarnDTO> overrunRecordWarnDTOList = overrunRecordDTOS.stream()
                .filter(transactionHandleHelp::checkEmpty)
                .filter(transactionHandleHelp::checkRequired)
                .filter(transactionHandleHelp::checkOverrunRecord)
                .map(transactionHandleHelp::toOverrunRecordWarnDTO)
                .filter(transactionHandleHelp::checkEmpty)
                .collect(Collectors.toList());
        return overrunRecordWarnDTOList;
    }

    /**
     * 实时超限数据推送
     * //此处进行统一阈值控制，波及范围：实时预警，站点值守，报警助手，手机APP，简版可视化
     *
     * @param overrunRecordWarnDTOList
     */
    @Override
    public void handleMessage(List<OverrunRecordWarnDTO> overrunRecordWarnDTOList) {
        List<RealTimeRecord> realTimeRecords = realTimeRecordAdapter.overrunRecordWarntoRealTimeRecords(overrunRecordWarnDTOList);
        List<RealTimeRecord> realTimeRecordsAfterThreshold = realTimeRecords.stream()
                .filter(ObjectUtil::isNotNull)
                .filter(overrunThresholdService::verifyData)
                .map(overrunThresholdService::isSevereOverrun).collect(Collectors.toList());
        if (CollUtil.isEmpty(realTimeRecordsAfterThreshold)) {
            return;
        }
        //实时预警，站点值守，报警助手，手机APP
        iMessageConsumerList.forEach(iMessageConsumer -> {
            executorService.execute(() -> {
                iMessageConsumer.receive(realTimeRecordsAfterThreshold);
            });
        });
    }
}

创新点及效果 ​

​ 本案例分析了MQ数据消费，设计实现一种基于模板模式的MQ消息对接处理方案，创新点及效果大致如下：

1、代码层次变得更加清晰，各个接口层**“分工明确”**；

2、面向接口编程，规范MQ数据消费流程，提供一种简洁统一的接口范式；

3、数据消费的行为由模板控制，业务实现处理接口即可，便于维护和扩展；

4、预警类数据统一解析、序列化、过滤再分发，实现**“一次消费，多次使用”**的目标，于此同时提高数据一致性；
