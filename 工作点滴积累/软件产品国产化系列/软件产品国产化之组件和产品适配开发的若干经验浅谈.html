前言 ​

在TOCC[1]国产化开发过程中记录了一些组件和产品升级的基本操作以及注意事项，还包括一些操作系统的疑难问题记录。或许可以给你带去一点帮助，在产品国产化的过程中少走一些弯路。

因为本次组件国产化未进行中间件的适配升级工作（即适配人大金仓和东方通），所以不会展开讲这两块东西。

硬件基础 ​

服务端环境1：arm+kylin

  • 单台服务器：（如多台服务器有差异需额外说明）

  • 内存：64GB及以上

  • 硬盘：500G以上

## 产品&组件版本规范

TOCC产品基于 “云远”构架_infosight V2.1.200进行集成开发，虽然我一直比较疑惑版本号的命名范式，但是据我的观察其中**“.200”的含义是ARM版本国产化操作系统的含义**；“.300”的含义是x86版国产化操作系统。但是也没有找到具体的资料证实这个事情，仅限于观察得知。

因此还是建议组件或者产品的版本号尽量跟着架构走，这样子会比较便于理解和版本对应维护。

所以TOCC产品的版本号为V2.0.200，两个自研业务组件toma[2]和vorm[3]同样是跟着架构的版本。

安装&启动脚本 ​

因为国产化操作系统也是基于Linux内核，一般来说组件的安装脚本postinstall.sh不需要做任何改动，即可在国产化环境完成安装。

而启动脚本方面，如果是通过Jsvc方式启动的，则需要替换调原本X86下的jsvc二进制文件ARM版本；如果是通过java方式启动，理论上则无需做任何改动。

接下来我贴上两种方式的启动脚本以及二进制文件，以做参考。

Jsvc方式启动

jsvc文件参考附件中"Jsvc_ARM"二进制文件，启动脚本（附件"__service.jsvc"）参考如下：

function do_exec()
{
    if [ ${_JRE_VERSION} -eq 8 ]
    then
    # JDK8
        Classpath_Extra_Jdk8=.:${JAVA_HOME}/lib/dt.jar:${JAVA_HOME}/lib/tools.jar
        _Classpath_jdk8=${Classpath_Extra_Jdk8}:"${_HOME_DIR}/config":"${_HOME_DIR}/lib/*":"${_HOME_DIR}/../../conf"
        $_JSVC -D${_ServerName} -home $JAVA_HOME -cwd ${_HOME_DIR}/config -umask 0011 -Djava.security.egd=file:/dev/./urandom -Djava.net.preferIPv4Stack=true -Dcom.sun.jndi.ldap.connect.pool.protocol="plain ssl" -XX:MetaspaceSize=${_MetaspaceSize} -XX:MaxMetaspaceSize=${_MaxMetaspaceSize} -Xms${_Xms} -Xmx${_Xmx}  -XX:CICompilerCount=2 -XX:+UseG1GC -XX:ConcGCThreads=1 -XX:ParallelGCThreads=4 -XX:+ExplicitGCInvokesConcurrent -XX:SurvivorRatio=8 -XX:+HeapDumpOnOutOfMemoryError -XX:+DisableExplicitGC -XX:+PrintGCDetails -XX:+PrintGCDateStamps -XX:+PrintGCTimeStamps -Xloggc:${_HOME_DIR}/logs/gc.log -XX:+UseGCLogFileRotation -XX:NumberOfGCLogFiles=10 -XX:GCLogFileSize=8M -Djava.io.tmpdir=${_HOME_DIR}/temp -Dloader.main=${_MainClass}  -wait 60 -pidfile $_Pid -outfile ${_LogOut} -errfile ${_LogErr} -cp ${_Classpath_jdk8} -procname ${_SystemctlName} $1 ${_StarCLASS}
    elif [ ${_JRE_VERSION} -eq 11 ]
    then
    # OPENJDK11
        _Classpath_Openjdk11=.:"${_HOME_DIR}/config":"${_HOME_DIR}/lib/*":"${_HOME_DIR}/../../conf":"${_HOME_DIR}/../../resource/resourcepath"
        $_JSVC -D${_ServerName} -home $JAVA_HOME -cwd ${_HOME_DIR}/config  -umask 0011 -Djava.security.egd=file:/dev/./urandom -Djava.net.preferIPv4Stack=true -Dcom.sun.jndi.ldap.connect.pool.protocol="plain ssl" -XX:MetaspaceSize=${_MetaspaceSize} -XX:MaxMetaspaceSize=${_MaxMetaspaceSize} -Xms${_Xms} -Xmx${_Xmx} -XX:SurvivorRatio=8 -XX:+HeapDumpOnOutOfMemoryError -XX:+DisableExplicitGC -Xlog:gc*:file=${_GCConfiguration} -Djava.io.tmpdir=${_HOME_DIR}/temp -Dloader.main=${_MainClass}  -wait 60 -pidfile $_Pid -outfile ${_LogOut} -errfile ${_LogErr} -cp ${_Classpath_Openjdk11}  -procname ${_SystemctlName} $1 ${_StarCLASS}
    else
        echo "Invalid JRE version"
    fi
}

# start the server
function start_cmd()
{
    echo "service name: $_ServerName starting......"
    chmod -R 777 ${_HOME_DIR}
    chmod -R 777 ${_HOME_DIR}/../../logs
    do_exec
    echo "${_ServerName} started"
}

case "$1" in
    start)
        if [ -f "$_Pid" ]; then
            # pid exists
            SERVICE_PID=`cat ${_Pid}`
            current_server_ruinning=`ps -ef |  grep -v grep | grep ${SERVICE_PID} | grep ${_ServerName} | wc -l`
            if [ ${current_server_ruinning} -eq 1 ]; then
                # pid is current service
                echo "service ${_ServerName} already running, don't need start again"
            else
                # pid isn't current service, delete the pid file
                echo "pid isn't belong to current service, delete the pid file and start current service"
                `rm -f ${_Pid}`
                start_cmd
            fi
        else
              # pid don't exists
              echo "pid don't exists, ready to start the service"
              start_cmd
        fi
            ;;
    stop)
        if [ -f "$_Pid" ]; then
            SERVICE_PID=`cat ${_Pid}`
            kill -15 ${SERVICE_PID}
        fi
           ;;
    restart)
        if [ -f "$_Pid" ]; then
            do_exec "-stop"
            do_exec
            echo "${_ServerName} restarted"
        else
            echo "service not running, will do nothing"
            exit 1
        fi
            ;;
    status)
        ps -ef | grep ${_ServerName}
        ;;
    *)
        echo "usage: service ${_ServerName} {start|stop|restart|status}" >&2
        exit 3
        ;;
esac

    # 判断是否为openjdk
    exists=`cat ${_JRE_VERSION_TXT} | grep 1.8 | wc -l`
    if [ ${exists} -gt 0 ]
    then
        _JRE_VERSION=8
    else
        _JRE_VERSION=11
    fi
}
judgeJreVersion
function do_exec()
{
    if [ ${_JRE_VERSION} -eq 8 ]
    then
    # JDK8
        nohup ${JAVA_BIN} $JAVA_OPTS_JDK8 ${_MainClass} >/dev/null 2>&1 &
    elif [ ${_JRE_VERSION} -eq 11 ]
    then
    # OPENJDK11
        nohup ${JAVA_BIN} $JAVA_OPTS_JDK11 ${_MainClass} >/dev/null 2>&1 &
    else
        echo "Invalid JRE version"
    fi
}
# 获取启动类名称(不含包名),用于jps命令获取pid
_MainClassName=${_MainClass##*.}
case "$1" in
    start)
        echo "service name: $_ServerName is starting..."
        chmod -R 777 ${_HOME_DIR}
        chmod -R 777 ${_HOME_DIR}/../../logs
        do_exec
            ;;
    stop)
        _Pid=`${JAVA_JPS} | grep ${_MainClassName} | awk '{print $1}'`
        echo "will kill pid ${_Pid}"
        kill -15 ${_Pid}
        echo ${_ServerName} is stopping
           ;;
    restart)
        _Pid=`${JAVA_JPS} | grep ${_MainClassName} | awk '{print $1}'`
        if [  X"$_Pid"==X ]; then
            kill -15 ${_Pid}
            do_exec
            echo "${_ServerName} restarted"
        else
            echo "service not running, will do nothing"
            exit 1
        fi
            ;;
    status)
        ps -ef | grep ${_MainClassName} | grep -v grep
        ;;
    *)
        echo "usage: service ${_ServerName} {start|stop|restart|status}" >&2
        exit 3
        ;;
esac

服务间依赖接口 ​

这个本身好像和国产化关系不是特别大，而是整个构架升级引起可能的接口变化，由于TOCC本身对其他组件依赖较小，所以暂未发现什么服务接口引发的问题。但是能够想到的一些资源同步，权限查询等接口可能存在适配工作量。

其中影响比较大的是由于构架升级到2.1.200，使用1.X打包的APP是无法使用的，需要基于2.X版本APP重新进行打包发布。

前端兼容性 ​

这里指的前端兼容性可以延伸理解为PC端国产化操作系统兼容，其中主要涉及的是：前端样式在国产化操作系统的样式兼容问题（比如蜂眼在奇安信浏览器V1.0.1969.2版本中就无法加载使用，提示‘请使用Chrome浏览器V70.0.0以上版本’）；插件助手的国产化适配（比如报警助手，下载器这块在TOCC中未做适配验证）；视频预览回放功能兼容（有极简播放器[4]和基于浏览器的H5Player[5]两个方案可选择，其中极简播放器需要安装相应客户端插件）;产品客户端适配（需要为相应ARM或者X86操作系统出相应的客户端包，可参考之后客户端打包章节）；

但是如果产品不用考虑PC端的国产化，那么理论上也就不需要做任何适配操作。

三方依赖 ​

指服务端依赖于二进制运行程序实现功能的一些依赖库，在TOCC中就遇到有：Libreoffice[6]文档转换（由于时间紧张，当时直接去掉了，不过对于在线文书或者报表，可选性方案有OnlyOffice[7]这类在线word预览引擎或者富文本编辑器TinyMCE[8]这类纯页面方案）和PhantomJS图表生成工具（这个花了一些时间找到了ARM版本，可参考 软件产品国产化之PhantomJS适配 wiki）。

注意事项 ​

  • 组件壳描述环境以及依赖相关的描述，尽量按照最小化原则进行配置，即没有计划进行兼容性测试的选项尽量不要选择；
  • 组件jc套件进行升级之后，注意检查xjob等一系列基础组件是否同步升级，内部依赖可能缺失commons-lang包；
  • 在解决开源合规性问题同时需要注意威盾组件安全扫描，两边目前未对齐，很可能解决了开源的问题，另外安全扫描会引发新问题，而且威盾的库一直在更新，每一轮都可能检出新问题；

防火墙配置 ​

产品封装过程中，特别需要注意的是防火墙问题，因为在自测阶段一般都会关闭防火墙，因此很难发现此方面可能存在的缺陷。

本次产品提测，主要遇到了两个由于防火墙引发的问题：consul组件的8068端口未开导致的组件寻址超时进而启动失败；vnsc组件的8604端口未开导致过流媒体录像失败。

若干服务疑难问题记录 ​

问题1：【麒麟server】断电重启之后，服务器无法连接 ​

问题分析：

重启操作系统导致系统的防火墙策略或者网卡配置发生重置，进而导致服务器直接无法ping通以及ssh连接。且之后打开防火墙即会复现此问题。

规避策略：

通过服务器管理页面进入系统，手动关闭防火墙后即可正常访问。另外或者参考系统差异性说明wiki“Kylin-v10-sp1上添加白名单导致防火墙异常的问题”章节升级firewalld防火墙。

问题2：【统信server】产品安装过程中，服务器ssh连接发生中断 ​

问题分析：

产品安装过程中，核心服务下发了防火墙策略导致系统的内置防火墙或者网卡配置发生变更，进而导致服务器直接无法ping通以及ssh连接。

规避策略：

通过服务器管理页面进入系统，重启操作系统以及手动关闭防火墙后即可正常访问。

问题3：【麒麟desktop】下载的视频文件，双击播放后，PC机系统卡死 ​

问题分析：

播放视频的工具为系统自带的“麒麟播放器”，并非必现，而是多次播放或者偶发性发生奔溃甚至导致系统卡死。因为测试环境系统为非授权激活版本且没有在线网络，初步猜测是播放器版本过低或自身兼容性缺陷导致。

规避策略：

到时候实际使用场景中升级最新的播放器版本或者下载其他第三方播放器，观察是否复现此问题。

问题4：【麒麟desktop&统信desktop】极简播放器在PC客户端重启之后未自启动问题 ​

问题分析：

极简播放器未加入系统自启动目录中，导致无法自启动。

规避策略：

编写将极简添加到自启动目录的执行脚本或者引导用户手动开启极简。

问题5：【统信desktop】无法预览下载的excel和word文件 ​

问题分析：

测试的统信PC机器未预装office软件。

规避策略：

实际使用时候，在正版激活的统信PC机中安装相应的office软件，如WPS。

问题6：【麒麟desktop&统信desktop】无法正常卸载客户端 ​

问题分析：

手动卸载客户端，显示卸载成功（通过命令’dpkg --get-selections|grep client’），但是实际客户端还能打开。初步判定是不是手动下发过客户端配置导致？？？

规避策略：

重启服务器之后再次进行卸载操作即可正常卸载。

结束语 ​

在国产化的过程中，除了本身国产化的适配，其实还需要面对各种安全扫描，开源合规性以及系统操作习惯的问题，在这些方面也花了很多心思和精力。

另外在进行轮次测试阶段，需要应对各种系统不稳定问题或者说操作习惯改变带来的不适感，遇到问题那就努力解决问题吧。在科技方面，我们要走的路其实还很长，还有很多地方需要努力去做的更好。
